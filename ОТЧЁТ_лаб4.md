## Цель работы

Реализовать простой HTTP прокси‑сервер на Socket API с поддержкой многопоточности и журналированием проксируемых запросов (URL и код ответа). Дополнительно реализовать фильтрацию сайтов по чёрному списку доменов и/или URL с возвратом страницы блокировки.

---

## Поддержка HTTP (без HTTPS)

Прокси поддерживает только HTTP‑запросы. HTTPS не требуется, поэтому метод `CONNECT` не проксируется — сервер возвращает `501 Not Implemented`.

---

## Особенность URI в запросах к прокси (absolute-form → origin-form)

При работе через прокси браузер отправляет request-line в виде полного URL (absolute-form), например:

`GET http://live.legendy.by:8000/legendyfm HTTP/1.1`

Однако сервер назначения часто ожидает запрос в виде пути (origin-form):

`GET /legendyfm HTTP/1.1`

Поэтому прокси парсит URL, выделяет `host:port` и `path`, и при пересылке upstream переписывает request-line в origin-form.

---

## Многопоточность и потоковая прокачка данных

Прокси принимает входящие TCP‑соединения и обрабатывает каждого клиента в отдельном потоке (`threading.Thread(..., daemon=True)`).

Ответ от сервера назначения проксируется **потоково** (без накопления целиком в памяти) — важно для длинных ответов/стриминга (например, онлайн‑радио).

Для упрощения определения конца ответа при пересылке upstream добавляется заголовок `Connection: close`, поэтому прокси читает ответ до закрытия соединения сервером назначения.

---

## Журналирование (URL и код ответа)

После получения заголовков ответа прокси извлекает код статуса (например, `200`, `404`) и выводит в консоль строку вида:

`[HH:MM:SS] <client_ip>:<client_port> -> <absolute_url> => <status_code>`

При блокировке по чёрному списку лог помечается как:

`=> 403 (BLOCKED)`

---

## Чёрный список (дополнительное задание)

Чёрный список задаётся в файле конфигурации `config.json`:

- `blocked_domains` — список доменов для блокировки (точное совпадение или по суффиксу).  
  Пример: `example.com` блокирует и `example.com`, и `a.example.com`.
- `blocked_urls` — список URL для блокировки по префиксу.  
  Пример: `http://live.legendy.by:8000/legendyfm`.

При попадании запроса под правила прокси возвращает:

- `HTTP/1.1 403 Forbidden`
- HTML‑страницу с сообщением об ошибке и указанием заблокированного адреса

## Запуск и настройка браузера

### Команда запуска

    python3 http_proxy_blacklist.py --listen-host 127.0.0.1 --listen-port 8080 --config config.json

### Настройка прокси в браузере

В браузере включается ручной прокси только для HTTP:

- HTTP proxy: `127.0.0.1`
- Port: `8080`

HTTPS можно не настраивать (по условию не требуется).

## Вывод

В ходе работы реализован многопоточный HTTP прокси‑сервер на Socket API, выполняющий пересылку HTTP‑запросов и ответов с журналированием URL и кодов ответа. Учтена особенность формата request-line при работе браузера через прокси (absolute-form) и выполнено преобразование в origin-form при передаче серверу назначения. Дополнительно реализована фильтрация по чёрному списку доменов и URL с возвратом предопределённой страницы блокировки и кода `403 Forbidden`.
